
S3 can host static websites, but serving them directly from S3 means slower load times for users far from your bucket's region and no HTTPS by default. CloudFront solves both: it caches your content at edge locations worldwide and serves everything over HTTPS with a free SSL certificate.

The combination is common—S3 for storage, CloudFront as the CDN. Setting it up requires coordinating bucket names, Origin Access Control (OAC), and bucket policies. You can manage this in one CloudFormation stack, but splitting it into two stacks teaches a pattern you'll use often: export values from one stack and import them into another.

This separation lets you deploy the S3 bucket first, upload content, then deploy CloudFront independently. It also makes each stack reusable—you can swap CloudFront distributions or point multiple distributions to the same bucket without touching the S3 stack.

Here's the path: create an S3 bucket stack that exports the bucket name, upload your files, then create a CloudFront stack that imports that name. CloudFormation handles the OAC and bucket policy automatically.

## What You'll Build

Two CloudFormation stacks. The first creates an S3 bucket with versioning, encryption, and public access blocked. It exports the bucket name and domain. The second stack creates a CloudFront distribution with Origin Access Control, imports the bucket name, and attaches the required bucket policy.

After deployment, you'll upload demo files from a GitHub repository, test the CloudFront endpoint, and verify caching behavior.

```
Stack 1: S3 Bucket
  ├─ Bucket with versioning and encryption
  ├─ Exports: BucketName, BucketDomainName
  └─ User uploads files via AWS CLI

Stack 2: CloudFront Distribution
  ├─ Imports: BucketName (from Stack 1)
  ├─ Origin Access Control
  ├─ CloudFront distribution
  └─ Auto-attached bucket policy
```

| Component | Purpose |
|-----------|---------|
| S3 Stack | Creates bucket, exports name for reuse |
| CloudFront Stack | Imports bucket name, creates distribution and OAC |
| Origin Access Control | Restricts S3 access to CloudFront only |
| Bucket Policy | Auto-generated by CloudFormation for OAC |

## Prerequisites

You'll need AWS CLI v2, CloudFormation permissions (S3, CloudFront, IAM), and an AWS account in **us-west-2**. The demo files are available in the [buildwithbrainyl/ccp repository](https://github.com/buildwithbrainyl/ccp/tree/main/builders-day/s3/demo_s3).

## Step-by-Step Playbook

### Step 1: Create the S3 Bucket Stack

This stack creates the bucket and exports its name so the CloudFront stack can reference it. Exporting values is how CloudFormation stacks share information across stack boundaries.

```yaml
# s3-bucket.yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: 'S3 bucket for static website hosting with CloudFront'

Parameters:
  BucketName:
    Type: String
    Description: 'Globally unique bucket name'
    Default: 'cloudfront-demo-bucket'

Resources:
  StaticWebsiteBucket:
    Type: AWS::S3::Bucket
    Properties:
      BucketName: !Ref BucketName
      VersioningConfiguration:
        Status: Enabled
      BucketEncryption:
        ServerSideEncryptionConfiguration:
          - ServerSideEncryptionByDefault:
              SSEAlgorithm: AES256
      PublicAccessBlockConfiguration:
        BlockPublicAcls: true
        BlockPublicPolicy: true
        IgnorePublicAcls: true
        RestrictPublicBuckets: true

Outputs:
  BucketName:
    Description: 'Name of the S3 bucket'
    Value: !Ref StaticWebsiteBucket
    Export:
      Name: !Sub '${AWS::StackName}-BucketName'

  BucketDomainName:
    Description: 'Domain name of the S3 bucket'
    Value: !GetAtt StaticWebsiteBucket.RegionalDomainName
    Export:
      Name: !Sub '${AWS::StackName}-BucketDomainName'
```

The `Export` section in the Outputs makes these values available to other stacks in the same account and region. The export name includes the stack name to avoid conflicts.

Deploy the stack:

```bash
aws cloudformation create-stack \
  --stack-name s3-static-website \
  --template-body file://s3-bucket.yaml \
  --parameters ParameterKey=BucketName,ParameterValue=cloudfront-demo-$(date +%Y%m%d) \
  --region us-west-2

# Wait for completion
aws cloudformation wait stack-create-complete \
  --stack-name s3-static-website \
  --region us-west-2
```

Get the bucket name from the outputs:

```bash
BUCKET_NAME=$(aws cloudformation describe-stacks \
  --stack-name s3-static-website \
  --query 'Stacks[0].Outputs[?OutputKey==`BucketName`].OutputValue' \
  --output text \
  --region us-west-2)

echo "Bucket name: $BUCKET_NAME"
```

### Step 3: Create the CloudFront Distribution Stack

This stack imports the bucket name from the S3 stack using `Fn::ImportValue`. It creates an Origin Access Control, CloudFront distribution, and automatically attaches the bucket policy that allows CloudFront to read objects.

```yaml
# cloudfront-distribution.yaml
AWSTemplateFormatVersion: '2010-09-09'
Description: 'CloudFront distribution for S3 static website'

Parameters:
  S3StackName:
    Type: String
    Description: 'Name of the S3 bucket stack'
    Default: 's3-static-website'

Resources:
  OriginAccessControl:
    Type: AWS::CloudFront::OriginAccessControl
    Properties:
      OriginAccessControlConfig:
        Name: !Sub '${AWS::StackName}-OAC'
        OriginAccessControlOriginType: s3
        SigningBehavior: always
        SigningProtocol: sigv4

  CloudFrontDistribution:
    Type: AWS::CloudFront::Distribution
    Properties:
      DistributionConfig:
        Origins:
          - Id: S3Origin
            DomainName:
              Fn::ImportValue:
                Fn::Sub: '${S3StackName}-BucketDomainName'
            OriginAccessControlId: !GetAtt OriginAccessControl.Id
            S3OriginConfig: {}
        Enabled: true
        DefaultRootObject: index.html
        Comment: 'Static website distribution'
        DefaultCacheBehavior:
          TargetOriginId: S3Origin
          ViewerProtocolPolicy: redirect-to-https
          AllowedMethods:
            - GET
            - HEAD
          CachedMethods:
            - GET
            - HEAD
          ForwardedValues:
            QueryString: false
            Cookies:
              Forward: none
          MinTTL: 0
          DefaultTTL: 86400
          MaxTTL: 31536000
          Compress: true
        PriceClass: PriceClass_All
        HttpVersion: http2
        IPV6Enabled: true
        CustomErrorResponses:
          - ErrorCode: 403
            ResponseCode: 404
            ResponsePagePath: /error.html
            ErrorCachingMinTTL: 300
          - ErrorCode: 404
            ResponseCode: 404
            ResponsePagePath: /error.html
            ErrorCachingMinTTL: 300

  BucketPolicy:
    Type: AWS::S3::BucketPolicy
    Properties:
      Bucket: !Fn::ImportValue
        !Sub '${S3StackName}-BucketName'
      PolicyDocument:
        Statement:
          - Sid: AllowCloudFrontServicePrincipal
            Effect: Allow
            Principal:
              Service: cloudfront.amazonaws.com
            Action: s3:GetObject
            Resource: !Sub
              - 'arn:aws:s3:::${BucketName}/*'
              - BucketName: !Fn::ImportValue
                  !Sub '${S3StackName}-BucketName'
            Condition:
              StringEquals:
                AWS:SourceArn: !Sub 'arn:aws:cloudfront::${AWS::AccountId}:distribution/${CloudFrontDistribution}'

Outputs:
  DistributionId:
    Description: 'CloudFront distribution ID'
    Value: !Ref CloudFrontDistribution

  DistributionDomainName:
    Description: 'CloudFront distribution domain name'
    Value: !GetAtt CloudFrontDistribution.DomainName

  DistributionURL:
    Description: 'Full URL to access the website'
    Value: !Sub 'https://${CloudFrontDistribution.DomainName}'
```

The `Fn::ImportValue` function retrieves exported values from the S3 stack. The bucket policy uses a condition to restrict access to requests from this specific CloudFront distribution.

Deploy the CloudFront stack:

```bash
aws cloudformation create-stack \
  --stack-name cloudfront-distribution \
  --template-body file://cloudfront-distribution.yaml \
  --parameters ParameterKey=S3StackName,ParameterValue=s3-static-website \
  --region us-west-2

# Wait for deployment (CloudFront takes 10-15 minutes)
aws cloudformation wait stack-create-complete \
  --stack-name cloudfront-distribution \
  --region us-west-2
```

Get the CloudFront domain name:

```bash
DISTRIBUTION_URL=$(aws cloudformation describe-stacks \
  --stack-name cloudfront-distribution \
  --query 'Stacks[0].Outputs[?OutputKey==`DistributionURL`].OutputValue' \
  --output text \
  --region us-west-2)

echo "CloudFront URL: $DISTRIBUTION_URL"
```

### Step 4: Understanding the Export/Import Pattern

This pattern separates concerns and enables reuse. The S3 stack owns the bucket lifecycle; the CloudFront stack owns the distribution. You can:

- Deploy multiple CloudFront distributions pointing to the same bucket
- Update the CloudFront stack without touching S3
- Delete and recreate CloudFront without affecting the bucket

The export names are scoped to the stack name, so multiple S3 stacks can coexist. Import values are resolved at stack creation time—if the export doesn't exist, CloudFormation fails with a clear error.

⚠️ **Caution:** You cannot delete a stack that exports values that are still imported by other stacks. Delete importing stacks first, then the exporting stack.

Open the CloudFront URL in your browser:

echo "Open in browser: $DISTRIBUTION_URL"
```

You should see the demo website. Test the error page:

echo "Test error page: $DISTRIBUTION_URL/nonexistent.html"
```

The custom error page should appear.

### Step 6: Test Cache Invalidation

Upload a modified version of `index.html`:

```bash
# Use v1/index.html downloaded in Step 2 (or download if needed)
# If you're in a new session, download it:
# mkdir -p v1 && curl -o v1/index.html https://raw.githubusercontent.com/buildwithbrainyl/ccp/main/builders-day/s3/demo_s3/v1/index.html

# Upload as index.html (replaces existing)
aws s3 cp v1/index.html s3://$BUCKET_NAME/index.html --region us-west-2
```

Refresh the CloudFront URL. The content may not change immediately due to caching. Create an invalidation:

```bash
DISTRIBUTION_ID=$(aws cloudformation describe-stacks \
  --stack-name cloudfront-distribution \
  --query 'Stacks[0].Outputs[?OutputKey==`DistributionId`].OutputValue' \
  --output text \
  --region us-west-2)

aws cloudfront create-invalidation \
  --distribution-id $DISTRIBUTION_ID \
  --paths "/index.html" \
  --region us-west-2
```

Wait 2-3 minutes, then refresh the browser. The updated content should appear.

## Validation

Verify both stacks are deployed:

```bash
aws cloudformation describe-stacks \
  --stack-name s3-static-website \
  --query 'Stacks[0].StackStatus' \
  --output text \
  --region us-west-2
# Expected: CREATE_COMPLETE

aws cloudformation describe-stacks \
  --stack-name cloudfront-distribution \
  --query 'Stacks[0].StackStatus' \
  --output text \
  --region us-west-2
# Expected: CREATE_COMPLETE
```

Check the exported values:

```bash
aws cloudformation list-exports \
  --query 'Exports[?contains(Name, `s3-static-website`)].{Name:Name,Value:Value}' \
  --output table \
  --region us-west-2
```

Verify the bucket policy was attached:

```bash
aws s3api get-bucket-policy \
  --bucket $BUCKET_NAME \
  --region us-west-2 \
  --query Policy \
  --output text | jq .
```

The policy should allow `cloudfront.amazonaws.com` to perform `s3:GetObject` with a condition matching the CloudFront distribution ARN.

## Cleanup

Delete stacks in reverse order—CloudFront first, then S3:

```bash
# Delete CloudFront stack
aws cloudformation delete-stack \
  --stack-name cloudfront-distribution \
  --region us-west-2

aws cloudformation wait stack-delete-complete \
  --stack-name cloudfront-distribution \
  --region us-west-2

# Empty S3 bucket before deleting stack (versioned buckets require deleting all versions)
BUCKET_NAME=$(aws cloudformation describe-stacks \
  --stack-name s3-static-website \
  --query 'Stacks[0].Outputs[?OutputKey==`BucketName`].OutputValue' \
  --output text \
  --region us-west-2)

# Delete all object versions
aws s3api delete-objects \
  --bucket $BUCKET_NAME \
  --delete "$(aws s3api list-object-versions --bucket $BUCKET_NAME --query '{Objects: Versions[].{Key:Key,VersionId:VersionId}}' --output json)" \
  --region us-west-2 2>/dev/null || true

# Delete delete markers
aws s3api delete-objects \
  --bucket $BUCKET_NAME \
  --delete "$(aws s3api list-object-versions --bucket $BUCKET_NAME --query '{Objects: DeleteMarkers[].{Key:Key,VersionId:VersionId}}' --output json)" \
  --region us-west-2 2>/dev/null || true

# Delete any remaining objects
aws s3 rm s3://$BUCKET_NAME --recursive --region us-west-2 2>/dev/null || true

# Delete S3 stack
aws cloudformation delete-stack \
  --stack-name s3-static-website \
  --region us-west-2

aws cloudformation wait stack-delete-complete \
  --stack-name s3-static-website \
  --region us-west-2
```

## Production Notes

**IAM Permissions:** Tighten the CloudFormation execution role to least privilege. Limit S3 actions to the specific bucket, and CloudFront actions to distribution management.

**Custom Domain:** Add an ACM certificate and Route 53 alias record. Use CloudFormation outputs to chain the certificate ARN into the distribution.

**Cache Policies:** Use managed cache policies (`CachingOptimized`, `CachingDisabled`) instead of `ForwardedValues` for better performance. The example uses `ForwardedValues` for clarity.

**Monitoring:** Enable CloudFront access logs to S3 and CloudWatch alarms for error rates. Add S3 bucket versioning alerts.

**Cost Optimization:** Use `PriceClass_100` (US, Canada, Europe) instead of `PriceClass_All` if your audience is regional. Enable Origin Shield to reduce origin requests and costs when serving high traffic or many edge locations.

**Multi-Stack Patterns:** This export/import pattern works across accounts using cross-account roles, but you'll need to pass values via SSM Parameter Store or Secrets Manager instead of CloudFormation exports.

See also: [Automate S3 and CloudFront Deployments with CodePipeline](./automate-s3-cloudfront-deployment-codepipeline.md)

## Conclusion

- Split S3 and CloudFront into separate stacks for better separation of concerns
- Use `Export` in outputs and `Fn::ImportValue` in templates to connect stacks
- CloudFormation automatically manages OAC and bucket policies when configured correctly
- Export/import fails fast if dependencies are missing, preventing broken deployments
- This pattern scales to complex multi-stack architectures
